# HardFault 排查指南

## 常见根因
- 空指针、野指针、函数指针异常。
- 数组越界、栈溢出、内存破坏。
- 访问未开启时钟的外设寄存器。
- 非法地址访问、未对齐访问。
- 时钟/Flash 配置错误导致运行异常。
- 中断栈或任务栈不足。
- RTOS 中在不安全上下文调用 API。

## 第一轮快速定位
1. 看是否稳定复现，定位大致触发场景。
2. 观察进入 HardFault 前最后执行到哪。
3. 如果能调试，查看调用栈、PC、LR、SP。
4. 检查最近新增代码：数组、指针、DMA、任务栈、中断。
5. 若改过主频、Flash wait state、链接脚本，优先回查。

## 推荐排查步骤

### 1. 缩小问题范围
- 注释最近新增模块，逐步恢复。
- 把复杂路径拆成最小可复现场景。
- 先确认是上电即 fault，还是运行一段时间后 fault。

### 2. 查栈与内存
- 检查主栈/任务栈大小。
- 检查大数组是否放在栈上。
- 检查 DMA 是否写坏内存。
- 检查 `memcpy`/`sprintf`/`printf` 等是否越界。

### 3. 查指针与寄存器访问
- 所有外设访问前是否已开时钟。
- 指针是否初始化。
- 结构体指针是否指向有效对象。
- 回调函数指针是否为空。

### 4. 查中断与 RTOS
- 中断优先级是否合理。
- 中断里是否调用了不安全的阻塞函数。
- RTOS 任务栈是否足够。
- 是否有优先级反转、死锁、堆耗尽。

## 实战建议
- 加入断言和统一错误处理。
- 为关键步骤加 GPIO 翻转或串口日志，定位死前最后位置。
- 使用 map 文件观察 RAM/Flash 占用。
- 若是 H7/F7，额外检查 Cache、一致性和对齐。
- 若是 FreeRTOS，检查 `uxTaskGetStackHighWaterMark()` 一类栈水位信息。

## 代码审查重点
- 中断与主循环共享变量是否安全。
- 是否把大对象放在局部变量里。
- 是否存在数组越界和长度未校验。
- 是否在 fault 前访问了空指针或非法地址。
- 是否有递归、深层调用、超大 `printf`。

## 回答此类问题时的优先输出
1. 先列最可能的 3~5 个根因。
2. 再给最短排查路径。
3. 最后根据用户芯片/工程结构补更具体建议。
